# Hamyar LLM Provider Adapter

Hamyar can optionally use a configured OpenAI-compatible LLM provider as a narrow helper behind the deterministic planner. The provider is off by default and is not required for the existing Hamyar flows.

## Runtime Contract

- The deterministic planner, Capability Registry, tenant-scoped tools, and live verification remain the source of truth.
- The LLM planner can only suggest one of the server-provided registry capabilities and relation paths.
- The LLM never executes tools, SQL, database queries, document lookup, file/image lookup, or write actions.
- Accepted planner output is converted into the existing static `business_search` plan shape before any data lookup runs.
- The answer polisher can only rewrite the already deterministic answer. It falls back to the deterministic answer if it introduces numbers, phones, dates, amounts, URLs, SQL, secrets, document links, or unsupported facts.
- Deterministic exact routes and option/code follow-ups win before any provider call.

## Configuration

The adapter uses Hamyar-specific environment variables only:

- `HAMYAR_LLM_ENABLED`
- `HAMYAR_LLM_PROVIDER`
- `HAMYAR_LLM_BASE_URL`
- `HAMYAR_LLM_API_KEY`
- `HAMYAR_LLM_MODEL`
- `HAMYAR_LLM_TIMEOUT_MS`
- `HAMYAR_LLM_MAX_TOKENS`
- `HAMYAR_LLM_TEMPERATURE`

`HAMYAR_LLM_ENABLED` must be exactly `true` before the adapter can make a provider request. If any required provider value is missing, the adapter fails closed and records a disabled reason in internal audit metadata.

Do not commit real values for these variables. `.env` files and provider keys remain deployment secrets.

## Provider Shape

The first adapter is OpenAI-compatible chat completions. `HAMYAR_LLM_BASE_URL` is configurable and is normalized as follows:

- A URL ending in `/chat/completions` is used as-is.
- A URL ending in `/v1` gets `/chat/completions` appended.
- Other base URLs get `/v1/chat/completions` appended.

No provider URL is hardcoded for a specific vendor.

## Safety Boundaries

The adapter rejects:

- SQL-like output and known table names
- secret/token/API-key-looking output
- URLs generated by the model
- code execution strings
- document/file/image lookup plans
- future write/action plans except as non-executable preview-only rejections
- unknown capabilities, unknown requested fields, and relation paths not present in the registry

The public status shape exposes only whether key material is configured. It does not expose the key.

## Evals

`node scripts/ai-agentic-context-eval.mjs` includes mock-provider coverage for:

- default-off behavior with zero provider calls
- valid planner output mapping back to static business search
- invalid JSON/schema fallback
- SQL rejection
- document lookup deferral
- future action preview-only rejection
- deterministic exact route precedence
- answer polishing fallback for hallucinated phone values

Dataset evals remain provider-free:

```bash
npm run hamyar:dataset:check
npm run hamyar:dataset:eval
```

## Rollback

Set `HAMYAR_LLM_ENABLED` to anything other than `true`, or remove the Hamyar provider variables. The deterministic Hamyar planner and existing tools continue to run.

No database migration is required for this adapter.
