# Phase 3 Public Tracking Lockdown Report

## Scope

Phase 3 locks public tracking and public document downloads to explicit customer-safe DTOs and server-side authorization checks. It keeps the existing Express/React/PostgreSQL architecture, local document storage, public tracking by token, public search, and authenticated tenant document behavior.

No routes, internal shipment/customer/document features, Persian/RTL behavior, Jalali display behavior, platform admin behavior, business data, or schema history were removed.

## Explicitly Forbidden Public Fields

Public tracking responses and document DTOs must not expose:

- `organizationId` / `organization_id`
- owner/user ids, assignee ids, staff internals
- user emails/phones
- password/session/token hashes
- raw tracking tokens
- document `storage_key`
- filesystem paths or storage paths
- private/internal workflow notes
- private blocker notes
- task internals
- raw `legacy_data`
- raw metadata
- billing/payment/invoice/receipt data
- SMS delivery data
- audit/change-log internals
- platform-admin data

`tests/e2e/helpers.ts` now scans public tracking JSON for forbidden key patterns and known forbidden serialized names.

## Document Visibility Rules

Public document listing and downloads now require:

- Tracking access is enabled on the shipment.
- The shipment is not archived.
- The document is in the same organization as the shipment.
- The document is explicitly `customer_visible`.
- The document is not archived.
- The document is linked directly to the tracked shipment, or is a customer-level document with no shipment link and the same tracked customer.
- Legacy shipments that still carry `legacy_data.customerId` are treated as customer-linked only when the referenced customer is in the same organization.
- Public JSON never includes `storage_key`; storage keys are used only after authorization inside the server-side streaming path.

Download authorization paths:

- `GET /api/public/track/:token/documents/:documentId` validates the tracking token hash and document linkage.
- `GET /api/public/documents/:id` now requires a signed `shipmentCode`, `expires`, and `signature` query tuple generated by the public DTO mapper. The signature is invalidated by tracking reset/disable because it is bound to the shipment tracking access update timestamp.

## Tracking Token Storage Findings

Current behavior:

- Generated tracking tokens use `crypto.randomBytes(32).toString("base64url")`.
- `customer_access_token_hash` stores a SHA-256 hash for lookup.
- `customer_access_token` also stores the raw token for compatibility with the existing internal UI that can retrieve/regenerate the customer link.
- Reset rotates both raw token and hash.
- Disable sets `customer_access_enabled = FALSE` but keeps the token values for compatibility.

Decision:

- Phase 3 does not run a raw-token cleanup migration because that could break the current internal link UI.
- Public DTOs no longer include raw tracking tokens in document URLs; they use signed document download URLs.

Future migration plan:

- Add a compatibility window where internal UI receives newly generated tokens only once.
- Stop storing `customer_access_token` for new links.
- Add a one-time admin-visible recovery/reset flow for old links.
- Backfill or verify `customer_access_token_hash` for every enabled public link.
- Remove raw token persistence after operational verification.

## Rate-Limit Findings

Existing public search and document downloads already use the app rate limiter.

Added:

- `GET /api/public/track/:token` now uses the same rate-limit plumbing with a dedicated `public-track-lookup` bucket.

Deferred to Phase 7:

- Tune public tracking/search/download thresholds for production traffic.
- Add alerting around high-volume invalid token/search/download attempts.
- Keep production `RATE_LIMIT_STORE=postgres`; do not introduce Redis in this phase.

## Remaining Risks

- Raw customer tracking tokens are still stored for internal UI compatibility.
- Signed public document URLs are short-lived and invalidated by access changes, but they are stateless; Phase 4/7 can consider a persisted public-document grant table if auditability is needed.
- Public company `contactText` can include organization contact details already present in the public tracking experience; review production copy and contact policy before broader rollout.
- Public route rate-limit thresholds should be tuned from production traffic data in Phase 7.

## Phase 4 Recommendation

Implement tracking-token raw-value retirement:

- Stop returning/storing raw tracking tokens except immediately after explicit generate/reset.
- Add an audited internal copy-link/reset workflow.
- Add migration checks for missing `customer_access_token_hash`.
- Add an operational dashboard for invalid public tracking/search/download attempts.
- Consider short-lived persisted public-document grants if document download audit trails become a compliance requirement.
